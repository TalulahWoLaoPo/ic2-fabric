# TODO: 唯一赠送装备（永不损坏）防刷方案讨论总结

> 状态：`TODO / 讨论结论记录`  
> 说明：本文档用于沉淀设计思路，**不代表必须实现**。

## 目标需求（讨论版）

1. 玩家进入服务器时发放一把专属稿子（或专属装备）。
2. 该物品永不损坏。
3. 同一玩家理论上只能持有 1 把有效专属物品，防止“刷第二把”。
4. 防止该赠送物品被当作合成材料，用于刷其他产物。

## 可行性结论

1. 在“普通玩家权限（非 OP / 非控制台）”场景下，可做到高强度防刷。
2. 在有管理权限（OP、控制台、NBT 编辑、回档）场景下，无法承诺绝对不可复制。
3. 方案应以“服务器权威 + 持久化唯一 ID + 事件拦截 + 低频巡检”组合实现，不建议单点方案。

## 已讨论的关键策略

## 1) 永不损坏

1. 可通过自定义物品逻辑实现（不消耗耐久或持续修复）。
2. 这部分实现复杂度低，风险较小。

## 2) 每人唯一（防第二把）

1. 给赠送物品写唯一标记（如 `ownerUUID + itemUUID + lockTag`）。
2. 玩家数据持久化记录“当前有效 itemUUID”。
3. 发现重复时只保留有效 ID，其余失效或删除。
4. 校验范围至少覆盖：
   - 玩家背包/盔甲栏/副手
   - 地上掉落物（`ItemEntity`）
   - 关键容器路径（按实际需求）

## 3) 丢进岩浆的处理策略

1. 不补发（防刷最强，体验最差）。
2. 可补发（推荐）：当“当前有效 ID”确认不存在时，重发并轮换新 ID，旧 ID 统一作废。
3. 自动回收（防丢）：进入危险销毁路径时回收给原主人。

## 4) 关于“枚举世界内所有 Item”

1. API 上可行：可遍历所有 `ServerWorld` 并扫描 `ItemEntity`。
2. 仅能扫描“已加载区块”。
3. 不应高频全量扫描，建议低频巡检（如 20~100 tick 级别）并配合事件拦截。

## 5) 关于 Mixin `ItemStack` / 全局构造拦截

1. 不建议作为主方案。
2. 原因：覆盖路径不完整、兼容性差、维护成本高、影响全局。
3. 若使用，也仅建议做轻量标记一致性检查，不承担主防线职责。

## 6) 关于在 `Item` 实例内做防重

1. 可行且更推荐（相对改 `ItemStack` 更可控）。
2. 需配合玩家级有效 ID、关键事件拦截和兜底巡检，不能只靠 `inventoryTick`。

## 7) 活塞推拉等典型复制问题

1. 关键方块设置不可推拉（`PistonBehavior.BLOCK` 或等价行为）。
2. 方块掉落逻辑做幂等，避免重复掉落。
3. 高价值物品仍需唯一 ID 体系兜底。

## 8) 防止赠送品作为合成材料

推荐双层：

1. 源头隔离（推荐）：做独立物品 ID（如 `gift_pickaxe`），不加入合成配方与相关材料标签。
2. 运行时拦截（兜底）：若输入包含赠送锁定标记（如 `ic2_gift_locked=true`），阻断产出。

## 建议落地顺序（如果未来要做）

1. 先做“独立赠送物品 ID + 不可合成”。
2. 再做“玩家有效 ID 持久化 + 登录发放/重发策略”。
3. 最后加“低频巡检 + 异常副本失效处理”。

## 非目标说明

1. 不承诺对 OP/控制台级行为的绝对防复制。
2. 不承诺对服务器回档导致的时序冲突做到零误报，仅可通过版本戳/轮换 ID 降低影响。
